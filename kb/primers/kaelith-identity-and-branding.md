# Kaelith Identity & Branding Primer

## What This Primer Is For

This primer exists to keep Kaelith projects consistent in structure, naming, identity, and organizational coherence.

It is not mainly about marketing polish.
It is about making sure projects:
- look like they belong in the Kaelith ecosystem
- are named and structured consistently
- can be tracked, installed, understood, and maintained across agents and workspaces
- do not drift into one-off local conventions that break discoverability

This is the one primer that is intentionally more identity/governance oriented than best-practice-heavy.

---

## Read This First

A project that works but does not fit the Kaelith ecosystem creates long-term friction.

Common problems this primer is meant to prevent:
- inconsistent naming
- weird local folder or file conventions
- unclear install/tracking structure
- style drift between projects
- documents or assets that do not fit shared expectations
- outputs that feel disconnected from the rest of the environment

The goal is not brand theater.
The goal is operational coherence.

---

## The 5–10 Rules To Not Violate

1. **Use Kaelith-consistent naming from the beginning.**  
   Renaming late creates avoidable confusion.

2. **Keep project structure legible to other agents.**  
   Do not invent local conventions without strong reason.

3. **Make install/tracking context explicit.**  
   Projects should be understandable in terms of where they live, how they are identified, and how they are maintained.

4. **Do not separate branding from system identity.**  
   Visual/style consistency and organizational consistency reinforce each other.

5. **Prefer repeatable structure over novelty.**  
   New formats should earn their complexity.

6. **Write titles, names, and summaries so they fit shared ecosystem expectations.**

7. **Keep cross-project references and ownership signals clear.**

8. **Avoid one-off formatting drift.**  
   Consistency helps search, onboarding, maintenance, and multi-agent work.

9. **If this project belongs in Kaelith, it should look and feel placeable within Kaelith.**

10. **Brand consistency is partly an operations problem, not just an aesthetic one.**

---

## Common Early Mistakes

- naming a project in a way that does not fit existing conventions
- inventing structure that only makes sense inside one workspace or one session
- failing to expose project identity clearly in documentation
- drifting into inconsistent titles, headings, or section styles across related docs
- treating brand consistency like surface polish instead of system coherence

---

## What To Think About Before You Start

### 1. Identity
Ask:
- what is this project called?
- does that name fit Kaelith conventions?
- is its identity obvious in docs and structure?

### 2. Placement
Ask:
- where does this project fit in the wider Kaelith ecosystem?
- what related systems, workspaces, or products should it look compatible with?

### 3. Consistency
Ask:
- are naming, file structure, and summaries aligned with shared expectations?
- what should stay consistent across similar projects?

### 4. Operational coherence
Ask:
- can another agent identify, install, track, and understand this cleanly?
- what conventions make later maintenance easier?

### 5. Documentation signals
Ask:
- is the project’s identity visible in README, metadata, and key docs?
- are ownership and scope easy to infer?

## Quick Routing Guide

This primer should be part of the universal baseline for nearly all projects.
It is especially important when:
- creating a new PM/project workspace
- naming or reshaping a project
- setting up catalogs, references, or reusable docs
- building systems meant to fit alongside other Kaelith projects

---

## Final Standard

Before a new project becomes structurally real, you should be able to say:

> This project fits Kaelith naming, structure, and identity conventions well enough that another agent can recognize, place, and maintain it without inventing extra translation layers.

If you cannot say that honestly, the project identity is not yet coherent.
